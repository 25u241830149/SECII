# CampusHub 数据库索引与隐私安全设计方案

为了满足 CampusHub 校园互助与交流平台的业务高可用、快速响应以及合规性（隐私保护）要求，我们摒弃传统的“有外键就加索引”的盲目思路，完全贴合业务的高频流转场景进行以下结构化设计。

---

## 第一部分：基于业务流转的高效索引设计

索引不是免费的，它会降低写入速度并增加存储成本。CampusHub 涉及大量并发写操作（如抢单、发消息、帖子点赞），因此索引设计应遵循：**优先保障高频列表查询、避免全表扫描、支持组合覆盖。**

### 1. 核心高频读场景（任务大厅与帖子瀑布流）

**互助任务表 (`t_task`)**

* **业务场景**：用户打开App/小程序进入任务大厅，默认拉取处于“开放中(OPEN)”状态、按发布时间倒序排列的任务。用户可能还会通过点击顶部的分类（快递、辅导等）进行筛选。
* **设计方案**：创建组合索引 `idx_status_category_time (status, category, create_time DESC)`。
* **为何这样设计**：这种最左前缀复合索引能直接支撑 `WHERE status = 'OPEN' AND category = 'DELIVERY' ORDER BY create_time DESC` 这样最高频的查询，并且避免了额外使用临时表进行 FileSort。

**论坛帖子表 (`t_post`)**

* **业务场景**：用户在论坛首页看帖，优先展示置顶帖、推荐帖，剩下的按时间和板块分类展示。
* **设计方案**：
  * 置顶/推荐流：`idx_priority_time (is_top, is_recommended, create_time DESC)`
  * 普通板块信息流：`idx_category_time (category, create_time DESC)`
* **为何这样设计**：论坛是典型的读多写少场景，且列表页必定带有分页与排序处理。针对核心查询模式定制组合索引可以极大提升并发读取能力并且减少慢查询。

### 2. 个人中心状态与流转场景（我的发布、我的接单）

**订单表 (`t_order`)**

* **业务场景**：用户进入“我的订单”，需要分别查看“我发布的（待完成、已完成）”和“我接单的（进行中、已完成）”。
* **设计方案**：
  * 作为发布者查询：`idx_poster_status (poster_id, status)`
  * 作为接单者查询：`idx_helper_status (helper_id, status)`
  * 订单与任务单点溯源：`idx_task_id (task_id)`（1对1关系找订单流转状态）
* **为何这样设计**：这类数据属于强账号隔离，第一查询条件永远是当前用户的ID，第二条件则是订单的流转状态标签（TabBar切换）。这种组合可以直接在B+树级别过滤当前页签需要的数据。

**站内信与聊天 (`t_message`, `t_chat_message`)**

* **业务场景1**：获取右上角的未读消息小红点数量（`SELECT count(*) FROM t_message WHERE receiver_id = ? AND is_read = false`）。
* **设计方案1**：`idx_receiver_read (receiver_id, is_read)`
* **业务场景2**：点开具体订单的聊天室，拉取该订单下的历史聊天记录排序。
* **设计方案2**：`idx_order_time (order_id, create_time DESC)`

### 3. 数据完整性与防并发写入

* **表 `u_user`**：`idx_student_id (student_id)`（唯一索引）。业务要求学号全局唯一，必须在数据库层面防并发注册相同发学号。
* **互动行为中间表 (`t_task_favorite`, `t_post_like`, `t_post_favorite`)**：
  * 必须建立级联唯一索引：如任务收藏表的 `uk_user_task (user_id, task_id)`。
  * **业务防抖要求**：前端极易出现因网络引起的狂点“点赞/收藏”按钮，导致短时间涌入大量重复请求。仅靠Redis或代码逻辑防不住幻读并发，必须通过唯一索引拦截，同时提供给后端以 `DuplicateKeyException` 捕获进行幂等处理依据。

## 第二部分：符合国家法律法规与校园管控的隐私数据处理

校园项目由于含有真实学籍数据，如果发生数据泄露，影响极坏。数据保护需采用“存储、传输、展示”三层隔离方案。

### 1. 密码数据安全：不可逆动态加盐

* **业务要求**：管理员和DBA绝对不能看到用户的原始密码。一旦数据库被脱库（拖库），黑客不能通过彩虹表撞库获取真实密码。
* **具体方案**：禁止使用 MD5 或 SHA-256 明文/静态盐加密。必须采用 `BCrypt` 或 `Argon2` 算法。
  * *原理*：`BCrypt` 每次执行都会生成一个随机的 Salt（盐），并和哈希结果保存在一起（例如 `$2a$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxep68oDD1`，其中包含算法类型、计算代价因子、随机盐以及密文）。即使两个用户的密码都是 `123456`，在数据库中的存储也是完全不一样的。

### 2. PII（个人身份信息）对称加密入库

在 `t_user_verification` (实名认证记录) 表中，会涉及 `real_name` (真实姓名) 或后续可能追加的 `phone_number` (手机号)、`id_card` (身份证号)。

* **数据存储介质保护**：这些信息在数据库中必须存储为**密文**。
* **加密选型**：由于业务功能可能需要解密使用（如接单后给双方展示经过脱敏的真实姓名增强信任感），或者管理员后台审核需要看明文，因此不能使用单向哈希，必须使用 **对称加密算法 (如 AES-256-GCM 或 SM4)**。
* **密钥管理 (KMS)**：加密密钥 `Data Key` 必须通过云服务 KMS 或独立服务进行托管，或放在服务端配置文件并严格通过 CI/CD Inject 注入，**绝对不能**与数据库存储在同一台服务器上。(即所谓的数据与钥匙分离)。

### 3. 用户主权类隐私（图片与证明证件）

* **学生证/一卡通照片 (`student_card_image`)**
  * *风险*：若直接提供包含文件真实存储地址的 URL（类似 `http://oss.../student_card/1.jpg`），恶意爬虫可以通过遍历文件名下载全校学生的证件信息。
  * *方案*：OSS (对象存储) 的 Bucket 权限严格设置为**私有读写 (Private)**。
  * 数据库表里 `student_card_image` 只保存 ObjectKey (即相对路径名)。
  * 当前端需要加载证件图像时（如审核员界面），由后端签发一个带过期时间的**临时预签名URL (Pre-signed URL)**。临时URL生命周期设置为 5 分钟，即便该链接被倒卖泄露，5分钟后自动失效无法被二次读取。

### 4. API接口数据脱敏（Data Masking）

* 后端所有涉及用户信息返回的接口，不能使用 `SELECT *` 退还整表实体 `User` 给前端。必须设计独立的返回层对象 (VO - View Object)。
* **防泄漏控制点**：前端调取接单人信息时，VO 里面应当只封存 `nickname`，剔除 `password`, `is_deleted` 等后台字段。若必须展示姓名和手机，执行动态脱敏，例如：接口向前端直接打出 `李*明`，手机号 `138****0921`。**不要把掩码工作交给前端做**，因为只要网络包可被抓取，数据已经实际泄露。
