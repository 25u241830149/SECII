## 第一条: AI使用原则
1. **AI 是辅助工具,工程师是最终责任主体**
   任何AI输出的内容(代码、文档、设计)在合入项目前必须经过理解、审查、测试。AI不替代团队成员的判断和决策。
2. **透明可追溯**
   所有AI 辅助产生的成果必须标注来源,保留Prompt日志,并记录在Git提交信息中。
3. **能力不退化**
   每个成员必须在每个编码阶段完成至少2个完全手写的核心函数(涉及业务关键逻辑、算法或安全相关),避免过度依赖AI导致基础能力下降。
4. **风险优先审查**
   高影响领域(安全、事务、金额、并发、权限)禁止直接使用AI生成未经人工严格审查的代码。

## 第二条: AI 使用范围

### 一、允许使用AI的场景
- **代码生成**
  - 实体类、DTO、Mapper接口、基础 CRUD Controller
  - 审查要求: 人工确认命名、字段类型正确
- **测试生成**
  - 单元测试骨架、Mock对象创建
  - 审查要求: 人工补充边界条件和断言
- **文档编写**
  - 需求初稿、API文档、README、注释
  - 审查要求: 人工修正业务细节
- **SQL 生成**
  - 建表语句、简单查询语句
  - 审查要求: 人工确认索引、事务隔离级别
- **解释与调试**
  - 解释错误堆栈、提供修改建议
  - 审查要求: 人工验证正确性
- **重构建议**
  - 提取重复代码、优化条件表达式
  - 审查要求: 人工评估影响范围

### 二、受限使用
以下场景AI 输出可作为“草稿”,但必须由非生成者的团队成员独立审查并测试后方可合入:
- **订单状态机:** 风险点 状态流转遗漏、非法转移。审查要求:画出状态图,人工模拟所有路径。
- **并发控制(抢单):** 风险点 乐观锁/悲观锁使用不当。审查要求:编写并发单元测试,验证无超接。
- **事务边界:** 风险点——多表操作未加事务。审查要求:检查 `@Transactional` 范围。
- **权限拦截器:** 风险点 越权风险。审查要求:编写测试用例验证不同角色访问。
- **金额/积分计算:** 风险点 溢出、精度丢失。审查要求:使用 `BigDecimal`,测试边界值。
- **敏感数据处理:** 风险点 密码、Token、身份证。审查要求:人工审查加密方式。
- **第三方接口调用:** 风险点——微信登录、支付。审查要求:模拟异常情况(超时、错误码)。

### 三、完全禁止使用AI的场景
1. **团队决策:** 架构选型、服务拆分、优先级排序、工作量评估。
2. **核心业务规则:** 订单取消条件、信用分计算公式(AI可提供参考公式,但最终规则由团队定)。
3. **安全敏感配置:** JWT密钥、数据库密码、API密钥(不得让AI生成或存储)。
4. **代码审查结论:** 不能由AI决定“是否通过PR”,必须由人工审查员做出最终判断。
5. **答辩准备:** 演讲稿、核心演示流程必须由团队自己设计。

## 第三条: 过程记录要求(可追溯性)

### 1. Prompt 日志
每次使用 AI 完成一个重要功能点(如一个接口、一个组件、一份文档)后,负责该任务的成员需在团队共享文档中填写一条结构化记录。

每个阶段结束时(P1~P6),团队必须从日常记录中筛选5~10条最具代表性的 Prompt 日志,整理成汇总文件,提交至 Git仓库。

**汇总文件规范:**
- **文件路径:** `docs/prompts/PX_代表性 Prompt日志.md` (X为阶段编号,如P1)
- **文件内容必须包含:**
  - 本阶段总 AI 对话次数
  - 筛选出的日志条目(每条按下方格式组织)
  - 每条日志附上对应的截图文件名(如有)

**单条日志汇总格式:**
- 日期
- 功能名称
- AI 工具
- Prompt 摘要(一句话说明提问内容)
- AI输出关键点(概括AI生成了什么,不必粘贴全部代码)
- 人工修改内容(具体改动点及原因)
- 发现的问题(AI的错误、遗漏或歧义)
- 产出文件路径
- 截图: `screenshots/XXX.png`(如有)

**截图规范:**
- **存放路径:** `docs/prompts/screenshots/`
- **命名格式:** `阶段_日期_功能描述.png`,例如 `P1_0409_登录接口.png`
- **内容要求:** 截图不必包含全部对话,但需清晰显示 Prompt 和 AI 的关键输出片段。

### 2. Git 提交规范
所有 commit message 必须标注来源,格式如下:
`<type>(<scope>): <subject> `

**示例:**
- `feat (user): add login API `
- `fix (order): resolve concurrent issue `
- `docs (readme): update setup guide `

**说明:**
- ``: 完全由AI生成,仅做了格式或微小调整。
- ``: 完全由人工编写。
- ``: AI生成后经过较大修改(需在PR描述中说明哪些部分修改了)。

### 3. 阶段反思日志
每个阶段(P1~P6)必须填写《AI协作反思日志》,回答:
- Q1: AI 哪里帮了我?附Prompt日志。
- Q2: AI哪里误导了我?如何发现和修正?
- Q3: 下次如何让AI更好地工作?

### 4. 代码注释要求
对于AI 生成的关键代码(如复杂算法、事务处理),应在方法注释中添加:
```java
/**
 * 本方法由 AI生成初稿,人工修改了异常处理部分。
 * AI Prompt: "生成接单服务的乐观锁实现”
 */
```

## 第四条: 代码合入规则
1. AI生成代码必须经过至少一名非作者的团队成员审查,审查者需在PR中明确写出“已审查AI生成部分,确认无逻辑错误”。
2. 每个编码阶段(P4、P5),每位成员必须提交至少2个完全手写的核心函数(不得由AI生成)。函数需满足:
   - 长度≥15行(不含注释和空行)
   - 涉及业务判断、算法或安全逻辑
   - 在PR中标记为 `[Human-Written]`
3. **PR描述模板(必须填写):**
   **变更类型:**
   - [ ] 新功能
   - [ ] Bug修复
   - [ ] 重构
   **AI 使用声明:**
   - 哪些代码/文档由AI生成:
   - 哪些部分人工修改:
   - 是否已保留Prompt日志: 是/否
   **测试情况:**
   - 单元测试覆盖: 是/否
   - 手动测试: 是/否
4. CI必须通过: 包括编译、单元测试(关键模块)、代码风格检查(如Checkstyle)。AI生成的代码若导致CI失败,由提交者负责修复。
5. 禁止直接 push到main分支。所有变更必须通过Pull Request,且至少一人批准。

## 第五条: 违规处理与监督机制

### 违规行为与处理措施
- **未标注[source]的AI生成代码合入**
  - **处理:** 发现后立即回退,由提交者在24小时内补充标注并重提PR。
- **使用AI生成受限场景代码(如事务)未经过专项审查**
  - **处理:** 删除该部分代码,要求人工重写或经过完整审查流程。
- **连续2次未保留 Prompt 日志**
  - **处理:** 团队内部通报,该成员需在下阶段额外提交一份“AI使用自省报告”。
- **故意使用AI生成完全禁止的內容(如安全配置)**
  - **处理:** 团队内部讨论,视情节取消该成员当阶段“主导”资格。
- **他人发现AI生成代码有明显缺陷而审查者未指出**
  - **处理:** 审查者和提交者共同承担责任,需补充测试用例。

### 监督方式
- 每周例会检查一次 `docs/prompts/` 目录的完整性。
- 随机抽查PR,验证 `[source]` 标注真实性。
- 阶段结束时,团队匿名投票评估契约遵守情况。
