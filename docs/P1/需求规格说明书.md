# 需求规格说明书

CampusHub

## 1. 引言

### 1.1 编写目的

本文档旨在详细说明“校园互助服务平台”的总体描述、功能需求与非功能需求。文档的预期读者包括项目发起人、开发团队、测试团队及运维人员。

### 1.2 项目范围

本项目是一个专为在校大学生设计的互助化社交与服务平台。平台涵盖快递代取、学习辅导、二手交易信息发布、校内组队匹配等“需求匹配和订单管理”的核心功能。经过调研，本项目决定在订单功能外另外实现校园论坛帖子发布、评论、互动等交流功能，打造一站式校园服务与社交生态。

### 1.3 定义

用户 (User): 指通过学号认证的在校学生。

发布者 (Poster): 发布互助需求的用户。

接单者 (Helper): 接受并履行互助任务的用户。

信用分 (Credit Score): 衡量用户在平台内诚信度的量化指标。

帖子(Post): 用户在校园论坛板块发布的各类校园相关内容，含图文、纯文字等形式。 评论(Comment): 用户针对论坛帖子发布的反馈、交流内容。

### 1.4 参考文献

IEEE Std 830-1998, IEEE Recommended Practice for Software Requirements Specifications.

## 2. 总体描述

### 2.1 产品视角

本系统为独立的 Web应用，通过校园网或移动互联网访问。系统需要与学校的学籍认证系统（模拟或接口对接）配合，以确保用户身份的真实性。

### 2.2 用户特征

在校学生: 主要用户群体。具备基本的智能设备操作能力，需求集中在节省时间（如代取快递）或资源共享（如二手书、组队）。

系统管理员: 具备计算机操作背景，负责维护平台秩序、处理申诉及管理违规账号。

### 2.3 约束条件

法律约束: 必须遵守校园管理规定及个人信息保护法。

技术约束: 前端需适配主流移动浏览器或作为小程序运行。

场景约束: 功能限定在校园地理范围内，不支持校外物流衔接。

### 2.4 假设与依赖

假设: 用户均拥有有效的校园身份证明。

依赖: 依赖稳定的校园网络环境及基础的邮件/短信发送服务。

## 4. 非功能需求

### 4.1 性能需求

响应时间: 关键页面（如首页、订单详情页）加载时间应小于 2 秒。[AI生成]

并发量: 系统需支持至少 100 名 用户同时在线操作而不出现崩溃。[AI生成]

数据库性能: 简单查询的响应时间应小于 500 毫秒。[AI生成]

### 4.2 安全性

身份校验: 敏感操作（如注销、修改密码）需进行二次验证。[AI生成]

隐私保护: 用户的学号、手机号除交易双方外不可见。[AI生成]

防御机制: 防止 SQL 注入及 XSS 跨站脚本攻击。[人工补充]

### 4.3 可用性

易用性: 界面应符合大学生审美，操作路径不超过 3 次点击。[AI生成+人工修改（次数）]

可用时间: 系统需保持 7×24 小时运行，年度故障停机时间不超过 48 小时。[AI生成]

### 4.4 可维护性

代码规范: 遵循标准的命名与架构规范。[AI生成]

日志记录: 系统需记录关键操作日志（登录、异常、高额订单变更）。[AI生成]
