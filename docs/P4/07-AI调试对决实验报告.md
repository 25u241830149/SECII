## 3. Bug 1：登录学号前后空格导致误判

### 3.1 问题描述

在 AI 直出登录校验代码后，登录主流程和现有测试可以通过，但人工审查发现一个边界问题：

```java
User user = userMapper.selectActiveByStudentId(request.studentId());
```

`validateLoginRequest` 只会拒绝 `null`、空字符串和全空格。对于 `" 20260001 "` 这种非空但带前后空格的输入，校验会通过，但数据库查询会按带空格的学号精确匹配，最终返回 `null`，并被误判为“学号或密码错误”。

### 3.2 纯人工调试过程

纯人工定位步骤：

1. 阅读 `AuthService.login`，确认登录前只调用 `validateLoginRequest`。
2. 阅读 `isBlank`，确认它只判断 `null` 或 `String.isBlank()`。
3. 阅读 `UserMapper.xml` 中的 `selectActiveByStudentId`，确认 SQL 是 `student_id = #{studentId}` 精确匹配。
4. 对照 P3 设计中“用户登录返回 token、密码安全、DTO 返回”等要求，确认该问题属于输入规整不足，不是设计要求冲突。
5. 得出修复方案：查询前对学号做 `trim()`，但不修改密码字段，避免改变密码语义。

人工定位耗时约 8 分钟。

### 3.3 AI 辅助调试过程

提供给 AI 的上下文摘要：

```text
登录逻辑中 validateLoginRequest 会拒绝空学号，但用户输入 " 20260001 " 时仍查询失败。
相关代码：
validateLoginRequest(request);
User user = userMapper.selectActiveByStudentId(request.studentId());
SQL 使用 student_id = #{studentId} 精确匹配。
请判断原因和最小修复方式。
```

AI 给出的定位：

- `isBlank()` 只处理全空白，不会自动去掉有效字符串两端空格。
- Mapper 是精确查询，因此带空格的学号不会命中真实用户。
- 最小修复是在查询前执行 `String studentId = request.studentId().trim();`。
- 不建议 trim 密码，因为密码中的空格可能是用户真实密码的一部分。

AI 辅助定位耗时约 2 分钟，定位准确，修复方案可用。

### 3.4 最终修复

修改文件：`backend/campushub-parent/campushub-user/src/main/java/com/campushub/user/service/AuthService.java`

```diff
 public LoginResponseDTO login(LoginRequest request) {
     validateLoginRequest(request);
-    User user = userMapper.selectActiveByStudentId(request.studentId());
+    String studentId = request.studentId().trim();
+    User user = userMapper.selectActiveByStudentId(studentId);
     if (user == null) {
         throw new BusinessException(ErrorCode.UNAUTHORIZED, "学号或密码错误");
     }
```

### 3.5 修复验证

执行命令：

```powershell
mvn -pl campushub-user -am '-Dtest=AuthServiceTest,AuthControllerTest,AuthControllerWebMvcTest' '-Dsurefire.failIfNoSpecifiedTests=false' test
```

验证结果：

- `AuthControllerTest`：2 个用例通过
- `AuthControllerWebMvcTest`：3 个用例通过
- `AuthServiceTest`：9 个用例通过
- 合计：14 个用例通过，0 failures，0 errors

## 4. Bug 2：Maven 测试命令在 PowerShell/Reactor 下失败

### 4.1 问题描述

为了运行登录相关现有测试，第一次执行命令：

```powershell
mvn -pl campushub-user -am -Dtest=AuthServiceTest,AuthControllerTest,AuthControllerWebMvcTest -Dit.test=AuthFlowIntegrationTest test
```

PowerShell 报错：

```text
参数列表中缺少参量。
```

将 `-Dtest` 加引号后继续执行，又出现 Maven Surefire 错误：

```text
No tests matching pattern "AuthServiceTest, AuthControllerTest, AuthControllerWebMvcTest" were executed!
```

原因是 `-am` 会同时构建 `campushub-common`，但指定的测试类只存在于 `campushub-user`，公共模块没有匹配测试，Surefire 默认因此失败。

### 4.2 纯人工调试过程

纯人工定位步骤：

1. 先根据 PowerShell 报错检查命令语法，发现逗号分隔的 `-Dtest` 参数需要整体加引号。
2. 加引号后再次运行，进入 Maven 构建阶段。
3. 阅读 Maven 输出，发现失败模块是 `campushub-common`，不是 `campushub-user`。
4. 结合 `-pl campushub-user -am` 的含义判断：`-am` 会构建依赖模块，因此公共模块也会执行 Surefire。
5. 确认公共模块没有 `AuthServiceTest` 等测试类，因此需要加 `-Dsurefire.failIfNoSpecifiedTests=false`。


### 4.3 AI 辅助调试过程

提供给 AI 的上下文摘要：

```text
我在 PowerShell 中执行 Maven：
mvn -pl campushub-user -am -Dtest=AuthServiceTest,AuthControllerTest,AuthControllerWebMvcTest test
先报“参数列表中缺少参量”。
给 -Dtest 加引号后，campushub-common 又报 No tests matching pattern。
请判断原因并给出可运行命令。
```

AI 给出的定位：

- PowerShell 会把未引用的逗号参数解析成参数列表，应把整个 `-Dtest=...` 用引号包起来。
- `-am` 会构建依赖模块，`campushub-common` 没有这些指定测试，因此 Surefire 报“没有匹配测试”。
- 可用命令是在引号基础上增加 `-Dsurefire.failIfNoSpecifiedTests=false`。

AI 辅助定位耗时约 3 分钟，定位准确，命令可用。

### 4.4 最终修复

最终采用命令：

```powershell
mvn -pl campushub-user -am '-Dtest=AuthServiceTest,AuthControllerTest,AuthControllerWebMvcTest' '-Dsurefire.failIfNoSpecifiedTests=false' test
```

验证结果：

```text
Tests run: 14, Failures: 0, Errors: 0, Skipped: 0
BUILD SUCCESS
```

补充说明：`AuthFlowIntegrationTest` 单独运行时因当前环境缺少可用 Docker/Testcontainers 环境失败，报错为 `Could not find a valid Docker environment`。该问题属于当前运行环境限制，不作为本次代码修复对象。

## 5. 分析问题

### 5.1 AI 表现较好的 Bug 类型

AI 在以下问题上表现较好：

- 报错信息明确、可从命令输出直接定位的问题。
- 局部代码逻辑清晰、上下文边界明确的问题。
- 需要组合解释工具链行为的问题，例如 PowerShell 参数解析、Maven Reactor、Surefire 测试选择器。
- 需要给出最小修复点的问题，例如只对 `studentId` trim，不动密码和 DTO。

### 5.2 AI 表现较差或需要人工把关的 Bug 类型

AI 仍需要人工把关的地方：

- 判断边界行为是否符合业务习惯，例如学号是否应该自动去除前后空格。
- 判断哪些失败属于代码问题，哪些属于环境问题，例如 Testcontainers 找不到 Docker。
- 判断修复是否会改变安全语义，例如不能随意 trim 密码。
- 判断是否需要补充测试，不能只因为现有测试通过就认为问题彻底消失。

### 5.3 AI 是否出现只修表象、不找根因

本次 AI 辅助调试没有明显只修表象：

- Bug 1 中，AI 没有建议放宽异常或改错误提示，而是定位到学号输入规整与 SQL 精确匹配之间的根因。
- Bug 2 中，AI 没有把 Maven 失败误判为测试代码失败，而是区分了 PowerShell 参数解析和 Reactor 多模块执行两个层面的原因。

但如果只提供“测试失败”而不给命令、模块路径和 Maven 输出，AI 很可能会泛泛建议“检查测试类名”或“清理缓存”，定位质量会下降。

### 5.4 人工提供哪些上下文后 AI 建议明显变好

以下上下文让 AI 的调试建议明显变好：

- 完整错误输出，而不是只说“运行失败”。
- 明确运行环境，例如 PowerShell、Maven 多模块、Spring Boot。
- 相关代码片段，例如 `AuthService.login`、`validateLoginRequest`、`UserMapper.xml`。
- 设计约束，例如 P3 要求密码不能明文、登录返回 DTO、注册和登录允许匿名访问。
- 当前目标，例如“不修改测试，只让现有测试通过”或“只做最小业务修复”。

## 6. 实验结论

本次 AI 调试对决说明：AI 在定位局部代码问题和解释工具链报错时能显著缩短时间，尤其适合把错误输出转化为可执行的验证命令和最小修复方案。但最终是否采纳 AI 方案，仍然必须由人工结合 P3 设计、业务语义和安全边界判断。

最终结果：

- Bug 1 已完成代码修复，登录相关现有测试通过。
- Bug 2 已得到可运行测试命令，登录相关现有测试通过。
- `AuthFlowIntegrationTest` 的 Docker/Testcontainers 限制已记录为环境问题，后续需要在可用 Docker 环境或 CI 中验证。
