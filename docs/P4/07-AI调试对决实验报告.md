# “AI 调试对决”实验报告

更新时间：2026-06-07

## 1. 实验说明

本报告选取 P4 交付物复核过程中真实出现的 2 个问题，对比“纯人工定位”和“AI 辅助定位”的差异。两个问题都直接影响 P4 交付质量：

- Bug 1：前端覆盖率测试失败，导致单元测试交付不能通过。
- Bug 2：P4 文档错误记录集成测试环境，导致交付说明与实际 CI/源码不一致。

## 3. Bug 1：前端公开文档页测试失败

### 3.1 问题描述

执行：

```powershell
cd frontend
npm.cmd run test:coverage
```

失败摘要：

```text
expected [ DOMWrapper, ... ] to have a length of 10 but got 14
expected wrapper.text() to contain '平台使用规则与双方责任说明'
```

### 3.2 纯人工调试过程

纯人工定位通常需要依次检查：

1. 是否是组件重复渲染导致目录数量翻倍。
2. 是否是 `publicDocuments.ts` 新增了帮助中心条目。
3. 注册页点击用户协议后打开的是哪个文档。
4. 测试应该改断言还是业务文案应该回退。

预计耗时约 10 分钟。

### 3.3 AI 辅助调试过程

AI 直接根据失败行号和实际输出判断：

- 帮助中心目录已从 10 个 section 扩展到 14 个。
- 服务条款弹窗当前文档标题为“用户服务协议”。
- 修复范围应限定在测试断言，而不是改业务组件。

定位耗时约 2 分钟。

### 3.4 最终修复

修改 `frontend/src/test/views/PublicDocumentView.test.ts`：

```diff
- expect(wrapper.findAll('.help-toc-link')).toHaveLength(10)
+ expect(wrapper.findAll('.help-toc-link')).toHaveLength(14)

- expect(wrapper.text()).toContain('平台使用规则与双方责任说明')
+ expect(wrapper.text()).toContain('用户服务协议')
```

验证结果：

- `npm.cmd run test:coverage` 通过
- 27 个测试文件、90 个用例全部通过
- 前端行覆盖率 81.67%

## 4. Bug 2：集成测试环境记录与真实实现不一致

### 4.1 问题描述

原 P4 文档中写到：

- 本机没有 Docker。
- 14 个集成测试全部被 Testcontainers 跳过。
- GitLab CI 通过 `docker:dind` 执行集成测试。

源码和当前环境复核后发现：

- `docker --version` 显示 Docker Desktop 可用。
- `docker ps` 显示 PostGIS 和 Redis 容器正在运行且 healthy。
- 本机已有 `postgis/postgis:16-3.4`、`redis:7-alpine`、`testcontainers/ryuk:0.12.0` 镜像。
- `.gitlab/backend.yml` 使用 GitLab service，不是 `docker:dind`。

### 4.2 纯人工调试过程

纯人工一般会从文档开始逐项追溯：

1. 查看 `.gitlab/backend.yml`。
2. 检查 Docker 是否可用。
3. 运行集成测试。
4. 汇总 surefire 报告。
5. 回改 04、05、08、09 等文档。

预计耗时约 15 分钟。

### 4.3 AI 辅助调试过程

AI 先从文档中的矛盾点切入：

- 文档说“无 Docker”，但仓库有 `docker-compose.yml` 且本机可能已有容器。
- 文档说“docker:dind”，但 `.gitlab/backend.yml` 应以实际 YAML 为准。
- 需要运行 `docker ps`、`docker compose ps` 和 `mvn "-Dtest=*IntegrationTest"` 验证，而不是继续沿用旧记录。

定位耗时约 5 分钟。

### 4.4 最终修复

文档修正为：

- 本机 Docker 可用。
- 后端集成测试 6 个类、16 个用例全部通过，0 跳过。
- GitLab 后端集成测试使用 PostGIS/Redis service，并设置 `CAMPUSHUB_TESTCONTAINERS_ENABLED=false`。
- 不再写“本机无 Docker”和“docker:dind”。

验证命令：

```powershell
cd backend/campushub-parent
mvn -q "-Dtest=*IntegrationTest" "-Dsurefire.failIfNoSpecifiedTests=false" "-DfailIfNoTests=false" test
```

验证结果：

- 6 个集成测试类通过
- 16 个用例通过
- 失败/错误/跳过均为 0

## 5. 对比分析

### AI 表现较好的 Bug 类型

- 测试失败信息清晰、行号明确的问题。
- 文档与源码/CI 配置矛盾的问题。
- 需要快速列出核验命令的问题。

### AI 表现较弱的 Bug 类型

- 需要判断课程评分偏好的问题，例如是否必须追加单方法端到端测试。
- 需要真实 GitLab Runner 页面结果的问题，因为本地仓库无法证明外部平台状态。
- 需要业务取舍的问题，例如账号注销是否要纳入 P4 主演示路径。

### 是否出现只修表象

Bug 1 没有只修表象，因为人工检查确认当前帮助中心确实有 14 个 section，且“用户服务协议”是当前真实文档标题。

Bug 2 没有只修表象，因为最终不仅改了文档表述，还实际运行了 Docker、集成测试和 surefire 汇总。

### 哪些上下文让 AI 调试建议变好

- 完整失败输出和失败行号。
- 当前源码路径，例如 `publicDocuments.ts`、`.gitlab/backend.yml`。
- 本机环境信息，例如 Docker 是否可用。
- 复测命令和最新 surefire/coverage 结果。

## 6. 结论

AI 在本次调试中最有价值的地方，是快速把“测试失败/文档矛盾”拆成可核验命令和最小修复点。最终方案仍需要人工执行命令、阅读源码并判断是否会误改业务行为。AI 的建议可以加速定位，但不能替代“以当前仓库事实为准”的复核。
