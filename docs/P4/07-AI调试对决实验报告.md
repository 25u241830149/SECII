# “AI 调试对决”实验报告

更新时间：2026-06-06

## 1. 实验说明

本报告基于 2026-06-06 整理 P4 交付物时发生的两次真实调试事件。为了满足课程“纯人工 vs AI 辅助”的对比要求，表中：

- `AI 辅助定位耗时`：来自本次实际处理过程
- `纯人工定位耗时`：基于同等上下文下的人工复盘估算

## 3. Bug 1 过程记录

### 问题描述

执行：

```powershell
mvn -Dtest=!*IntegrationTest -Dsurefire.failIfNoSpecifiedTests=false -DfailIfNoTests=false test
```

报错：

```text
Unknown lifecycle phase ".failIfNoSpecifiedTests=false"
```

### 纯人工调试思路

- 先怀疑 Maven 参数顺序错误
- 再检查是否是 Surefire 插件版本问题
- 最后才会联想到 PowerShell 参数转义

### AI 辅助调试思路

- 直接从报错文本识别这是“shell 改写参数”而不是 Maven 语义错误
- 快速给出 `--%` 方案

### 最终修复

```powershell
mvn --% -Dtest=!*IntegrationTest -Dsurefire.failIfNoSpecifiedTests=false -DfailIfNoTests=false test
```

## 4. Bug 2 过程记录

### 问题描述

我在同一时间并行执行：

- `mvn -DskipTests package`
- `mvn --% -Dtest=*IntegrationTest ... test`

随后 `campushub-order` 出现大批 `dto 不存在 / service 不存在 / symbol not found` 错误。

### 纯人工调试思路

- 先怀疑仓库本身代码损坏
- 再检查 `pom.xml`、模块依赖和源码路径
- 读完主测试文件与主代码后，才能想到是并行写 `target/` 的问题

### AI 辅助调试思路

- 先对比“刚刚单元测试明明通过”与“现在 package 失败”这一矛盾
- 据此快速排除源码本身缺失
- 锁定根因：两个 Maven 进程同时写同一模块的 `target/`

### 最终修复

- 取消并行执行
- 单独重跑 `mvn -DskipTests package`
- 后端打包成功

## 5. 对比分析

### AI 表现较好的 Bug 类型

- 报错文本非常明确的 shell/构建系统问题
- 能通过“前后结果矛盾”快速定位根因的工程问题
- CI、本地命令、测试执行链路中的环境问题

### AI 表现较弱的 Bug 类型

- 需要真实业务背景才能判断优先级的产品问题
- 需要团队结合演示目标做权衡的取舍问题
- 需要数据库真实数据或线上流量上下文的问题

### 是否出现“只修表象、不找根因”

这次没有。两次问题都找到了更底层的根因：

- Bug 1 根因是 shell 参数解释，不是 Maven 插件
- Bug 2 根因是并发写工作区，不是 `campushub-order` 代码缺失

### 哪些上下文让 AI 建议更好

- 给出完整命令，而不是只给错误摘要
- 说明当前 shell 是 PowerShell
- 说明两个命令是否在并行执行
- 提供“此前同仓库测试通过”的对照事实

## 6. 结论

这次“AI 调试对决”的结论比较明确：

- AI 在工程化、构建链、CI、脚本类问题上帮助非常大
- 但要得到稳定答案，必须把环境上下文一次性交代清楚
- 团队仍需人工决定“该不该并行跑”“哪条链路是验收主线”这类工程判断
