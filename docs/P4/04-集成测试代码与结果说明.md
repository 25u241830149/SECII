## 4. 已覆盖的正常流程

虽然仓库没有把“注册 -> 登录 -> 发布 -> 抢单 -> 完成 -> 评价”写成单个超长集成测试，但核心正常流程已经拆分到多个模块测试中：

| 流程环节 | 集成测试证据 |
| --- | --- |
| 注册 | `registerPersistsUserAndPendingVerification` |
| 登录 | `loginReturnsJwtAndUserPayload` |
| 发布任务 | `createDetailUpdateAndDeleteTaskThroughHttpFlow` 中的创建部分 |
| 抢单 | `grabAllowsOnlyOneWinnerAcrossFiftyConcurrentHelpers` |
| 完成/状态推进 | 订单状态完整规则主要由 `OrderStatusServiceTest` 保障 |
| 评价 | `createReviewWritesReviewAndCreditRecord` |
| 消息通知 | `notifyMarkReadAndDeleteFlowPersists`、`createUpdateAndDeleteNoticeFlowPersists` |

结论：

- 当前仓库采用“跨模块拆分”的集成测试策略，而不是单个全链路大用例
- 课程验收时可以接受这套策略，但若老师强制要求“一条龙单测脚本”，建议后补一个总流程测试
