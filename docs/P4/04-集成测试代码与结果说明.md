# 集成测试代码与结果说明

更新时间：2026-06-07

## 1. P4 要求对照

P4 要求集成测试至少覆盖：

- 1 个完整正常流程：用户注册、登录、发布需求、另一用户接单、完成、评价。
- 2 个异常流程：如未登录访问、重复接单、无效参数、越权操作等。
- 前后端联调中的关键接口调用。

当前仓库采用“按模块拆分、跨模块串联证据”的集成测试策略：不把全部业务动作写进一个超长测试方法，而是用 6 个集成测试类覆盖核心链路的关键环节，并用单元/控制器测试补充异常路径。

## 3. 集成测试环境

集成测试基类为：

```text
backend/campushub-parent/campushub-common/src/test/java/com/campushub/common/testing/ContainerizedIntegrationTestSupport.java
```

当前实现支持两种模式：

- 默认模式：`CAMPUSHUB_TESTCONTAINERS_ENABLED=true`，使用 Testcontainers 启动 `postgis/postgis:16-3.4` 和 `redis:7-alpine`。
- CI 外部服务模式：`.gitlab/backend.yml` 将 `CAMPUSHUB_TESTCONTAINERS_ENABLED=false`，并连接 GitLab service 中的 PostGIS/Redis。

本机 2026-06-07 验证时，Docker Desktop 可用，Testcontainers 成功启动临时 PostGIS 与 Redis 容器。

## 5. 正常流程覆盖说明

P4 指定的完整正常流程在当前测试中由多个集成测试类共同覆盖：

| 正常流程环节 | 集成测试证据 |
| --- | --- |
| 用户注册 | `registerPersistsUserAndPendingVerification` |
| 用户登录 | `loginReturnsJwtAndUserPayload` |
| 发布需求 | `createDetailUpdateAndDeleteTaskThroughHttpFlow` 中的任务创建 |
| 另一用户接单 | `grabAllowsOnlyOneWinnerAcrossFiftyConcurrentHelpers` 与抢单服务持久化断言 |
| 发布者确认/完成 | 订单状态完整规则由 `OrderStatusServiceTest` 覆盖；集成层重点覆盖抢单并发和持久化 |
| 完成后评价 | `createReviewWritesReviewAndCreditRecord` |
| 信用联动 | `createReviewWritesReviewAndCreditRecord` 验证 `t_credit_record` 与信用分变化 |
| 消息/公告联动 | `notifyMarkReadAndDeleteFlowPersists`、`createUpdateAndDeleteNoticeFlowPersists` |

说明：当前仓库尚未把上述所有步骤写成单个端到端测试方法。如果课程验收必须看到“一条龙”测试方法，可以在现有 6 个集成测试基础上追加一个总流程测试；但从覆盖内容看，核心正常流程的关键环节已经被集成层和服务层共同覆盖。

## 6. 异常流程覆盖说明

| 异常/边界场景 | 测试证据 |
| --- | --- |
| 重复抢单 | `grabRejectsDuplicateSubmissionFromSameHelper` |
| 发布者抢自己任务 | `grabRejectsPublisherGrabbingOwnTask` |
| 非开放任务不可抢 | `grabRejectsConcurrentRequestsWhenTaskIsNotOpen` |
| 并发抢单只允许一个赢家 | `grabAllowsOnlyOneWinnerAcrossFiftyConcurrentHelpers` |
| 下游失败后恢复任务状态 | `grabReopensTaskWhenDownstreamStepFailsAfterInsert` |
| 任务更新/删除权限与状态 | `TaskServiceTest`、`TaskControllerWebMvcTest` |
| 评价必须在订单完成后进行 | `ReviewServiceTest` |
| 管理接口权限与输入校验 | 各模块 WebMvc 控制器测试 |

## 7. 前后端联调接口覆盖

前端 `src/test/api/modules.test.ts` 覆盖了主要 API 封装路径，后端控制器和集成测试覆盖对应接口。重点接口包括：

| 业务 | 前端 API | 后端接口 |
| --- | --- | --- |
| 登录注册 | `/user/register`、`/user/login` | `AuthController` |
| 任务 | `/tasks`、`/tasks/{id}`、`/tasks/{id}/favorite`、`/tasks/{id}/comments` | `TaskController`、`TaskCommentController` |
| 订单 | `/orders/grab`、`/orders/{id}/confirm`、`/orders/{id}/complete`、`/orders/{id}/cancel` | `OrderController` |
| 评价 | `/reviews`、`/reviews/user/{id}`、`/reviews/order/{id}` | `ReviewController` |
| 消息 | `/messages`、`/messages/unread-count`、`/orders/{id}/chat` | `MessageController`、`ChatController` |
| 举报/后台 | `/reports`、`/admin/reports`、`/admin/stats/dashboard` | `ReportController`、`AdminStatsController` |

## 8. 结论

集成测试代码交付物已满足 P4 的主体要求：6 个集成测试类、16 个用例在本机 Docker/Testcontainers 环境中全部通过，覆盖注册登录、任务、抢单并发、评价信用、举报和消息公告。唯一需要人工决定的是是否为了展示形式再新增一个单方法端到端测试；当前文档不把尚未存在的单方法“一条龙测试”写成已实现。
