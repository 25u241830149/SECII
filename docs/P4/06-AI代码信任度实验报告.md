## 1. 实验对象

本次实验选择 P4 关键实验一建议的功能点：**用户登录校验**。

- 所属模块：`campushub-user`
- 核心文件：`backend/campushub-parent/campushub-user/src/main/java/com/campushub/user/service/AuthService.java`
- 核心方法：`AuthService.login(LoginRequest request)`
- 选择原因：登录校验属于 P0 用户与认证能力，逻辑规模适中，包含参数校验、密码校验、账号状态校验和 token 返回，适合观察 AI 生成代码能否直接落地。

## 3. AI 直出结果摘要

AI 直出代码补全了 `AuthService.login` 的核心流程：

1. 调用 `validateLoginRequest(request)` 检查请求对象、学号和密码是否为空。
2. 调用 `userMapper.selectActiveByStudentId(request.studentId())` 查询未逻辑删除用户。
3. 用户不存在或密码不匹配时抛出 `ErrorCode.UNAUTHORIZED`。
4. 用户状态为封禁时抛出 `ErrorCode.FORBIDDEN`。
5. 登录成功后调用 `jwtTokenProvider.generateAccessToken(...)` 生成 token，并返回 `LoginResponseDTO`。

AI 没有新增或修改测试代码，符合本次实验约束。

## 6. 主要问题与人工修复

| 序号 | 问题                                                                               | 影响                                                                         | 修复                                                                                  |
| ---- | ---------------------------------------------------------------------------------- | ---------------------------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| 1    | AI 直出代码直接使用 `request.studentId()` 查询用户，只校验全空格，不处理前后空格 | 用户输入 `" 20260001 "` 时，校验通过但查询不到用户，被误判为账号或密码错误 | 在查询前增加 `String studentId = request.studentId().trim();`，使用规整后的学号查询 |

人工修复代码片段：

```java
public LoginResponseDTO login(LoginRequest request) {
    validateLoginRequest(request);
    String studentId = request.studentId().trim();
    User user = userMapper.selectActiveByStudentId(studentId);
    if (user == null) {
        throw new BusinessException(ErrorCode.UNAUTHORIZED, "学号或密码错误");
    }
    if (!EncryptUtils.matchesPassword(request.password(), user.getPassword())) {
        throw new BusinessException(ErrorCode.UNAUTHORIZED, "学号或密码错误");
    }
    if (Integer.valueOf(USER_STATUS_BANNED).equals(user.getStatus())) {
        throw new BusinessException(ErrorCode.FORBIDDEN, "账号已被封禁");
    }

    UserRole role = UserDtoAssembler.roleFromCode(user.getRole());
    String token = jwtTokenProvider.generateAccessToken(user.getId(), user.getStudentId(), role);
    UserVerification verification = verificationMapper.selectLatestByUserId(user.getId());
    return new LoginResponseDTO(token, UserDtoAssembler.toUserInfo(user, verification));
}
```

## 9. 信任度分析

AI 直出代码在登录校验主路径上表现较好，能够正确使用项目已有的 `BusinessException`、`ErrorCode`、`EncryptUtils`、`JwtTokenProvider` 和 DTO 结构，且没有引入新框架或复杂设计。现有登录相关测试可以直接通过，说明 AI 对明确、局部、已有上下文充分的业务逻辑具备较高可用性。

不过测试发现的问题不是编译错误或测试失败，而是实际用户输入中的边界体验问题。现有测试没有覆盖带前后空格的学号，因此如果只看测试结果，容易高估 AI 代码的可靠性。
