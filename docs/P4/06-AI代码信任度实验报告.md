# “AI 代码信任度实验”报告

更新时间：2026-06-06

## 1. 实验对象

本次实验没有选业务接口本身，而是选择了一个对 P4 交付同样关键的“代码型产物”：

- 对象：Windows PowerShell 下执行后端非集成测试的 Maven 命令
- 原因：P4 的测试交付和 CI/CD 验证都依赖这条命令是否可直接运行

## 2. 实验 Prompt

> 请给出一条在 Windows PowerShell 中执行 CampusHub 后端“非集成测试”的 Maven 命令，要求排除 `*IntegrationTest`，并在没有匹配测试时不要失败。

## 3. AI 直出结果

AI 直出命令：

```powershell
mvn -Dtest=!*IntegrationTest -Dsurefire.failIfNoSpecifiedTests=false -DfailIfNoTests=false test
```

## 4. AI 直出运行结果

直接运行后，Maven 报错：

```text
Unknown lifecycle phase ".failIfNoSpecifiedTests=false"
```

问题并不在项目代码，而在 PowerShell 对参数的解释方式：

- `!` 所在的参数在 PowerShell 中存在额外解析风险
- 直接照抄 AI 结果会把后续参数拆坏

## 5. 人工审查发现的问题

| 序号 | 问题 | 如何发现 | 结论 |
| --- | --- | --- | --- |
| 1 | AI 只给了 Maven 参数，没有考虑 PowerShell 参数转义 | 命令立即报 `Unknown lifecycle phase` | 不可直接用 |
| 2 | AI 命令看起来“像正确 Maven 命令”，但不适配当前 shell | 同一仓库在 GitLab CI 中命令能用，本地 PowerShell 不能用 | 需要结合执行环境修正 |
| 3 | AI 没有提示 `npm`/`mvn` 在 Windows 下的 shell 差异 | 复盘本地运行过程时发现 | 缺少环境上下文会降低可用性 |

## 6. 人工修复方案

修复后的命令：

```powershell
mvn --% -Dtest=!*IntegrationTest -Dsurefire.failIfNoSpecifiedTests=false -DfailIfNoTests=false test
```

说明：

- `--%` 用于告诉 PowerShell 停止继续解释后续参数
- 修复后 Maven 能正确收到 `-Dtest=!*IntegrationTest`

## 7. 修复后运行结果

执行时间：2026-06-06

结果：

- Maven 运行成功
- 非集成测试执行完成
- 283 个测试通过

## 8. 实验对比表

| 指标 | AI 直出 | 人工审查修复后 |
| --- | --- | --- |
| 命令解析是否通过 | 否 | 是 |
| 功能是否可运行 | 否 | 是 |
| 测试是否通过 | 否 | 是，283 个测试通过 |
| 主要问题/修复说明 | 忽略了 PowerShell 参数转义 | 加入 `--%`，命令可稳定执行 |

## 9. 实验结论

本次实验说明：

- AI 在“生成看起来正确的命令/配置”上很快
- 但如果 Prompt 没有明确 shell 环境，AI 容易忽略平台差异
- 对于构建脚本、CI 命令、部署命令这类“环境敏感代码”，人工审查仍然是必须步骤

## 10. 结论评分

| 维度 | 评价 |
| --- | --- |
| AI 初稿速度 | 高 |
| AI 初稿可直接使用程度 | 低 |
| 人工修复成本 | 低 |
| 对团队的实际价值 | 中高，适合产出初稿，不适合免审直接执行 |
