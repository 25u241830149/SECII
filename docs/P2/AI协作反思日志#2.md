## 2. AI 助力分析（Q1）

**本阶段 AI 最大的贡献是：**
极大地加速了复杂技术方案的对比工作。AI 能够迅速拉齐不同架构（如微服务、事件驱动主架构、模块化单体）在“开发复杂度、可扩展性、团队学习成本”等多维度的优劣势，并直接生成结构化的对比表，为团队决策提供了丰富的理论依据。

**不同 Prompt 策略下 AI 回答质量的差异：**
当 Prompt 中仅提供“帮我设计一个校园互助系统架构”时，AI 倾向于推荐堆砌前沿技术的“大而全”体系（如完整的微服务+复杂的消息中间件）；但在 Prompt 中明确给出约束条件（如：“4人团队”、“10周开发周期”、“100人同时在线目标”）后，AI 生成的回答更加务实，能够针对性地分析架构的可行性与落地风险。

## 3. AI 误导分析（Q2）

**AI 在架构建议中的典型问题：**

- **问题1：** 盲目推荐过度设计（Over-engineering）。AI 最初建议将用户、订单、论坛等全部拆分为独立微服务，并引入完整的事件驱动机制，忽略了项目的真实体量范围。
- **如何发现：** 在评估团队 DevOps 能力及10周交付压力时，发现如果要处理分布式事务（尤其是抢单、取消订单等存在强一致性要求的核心链路）、服务发现和链路追踪，核心业务开发时间将被严重挤压。
- **如何修正：** 人工推翻了 AI 最初的主架构建议，重新划定了架构边界。将架构修正为“模块化单体 + MVC/分层”，并且只在异步通知（如订单完成预警、信用分变更）等非核心链路局部保留事件机制。

## 4. Prompt 改进计划（Q3）

**上阶段（P1）提出的改进措施，验证结果：**
在 P1 阶段提出了“在 Prompt 中强制要求结合具体工具栈（如 Vue3、Spring Boot 3）进行输出”的策略，在本阶段得到了有效验证。这使得生成的架构分析直接贴合团队的实际技术路线，减少了泛泛而谈的内容。

**本阶段新的改进措施：**
在即将进入的详细设计与代码生成阶段，Prompt 必须明确**上下文约束与边界**。对于特定模块的 Prompt，需要预先定义好 DTO/Entity 以及数据库事务的流转规则，制止 AI 随意创建额外的表或引入未经确认的三方组件。

## 5. 本阶段的核心工程判断

**决策内容：**
坚决放弃微服务和纯事件驱动架构，确立 **“前后端分离 + 模块化单体 + 后端 MVC/分层架构 + 局部事件机制预留”** 为最终系统架构，并敲定 MyBatis-Plus, Redis, PostgreSQL 的数据层基建。

**为什么 AI 无法做这个决策：**
AI 能够客观罗列技术的优缺点理论，但**架构本质上是团队资源的折衷与妥协**。AI 缺乏对 4 名学生团队实际工程能力、沟通成本、排错时间以及交付期限的切身体感。抢单和订单连贯操作能否通过单体数据库解决，取决于业务对最终交付的底线要求。这类涉及到“能不能按时把项目做完”的风险评估体系，是必须由人类工程师基于真实环境来主导决策的。
